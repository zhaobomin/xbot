# XBOT Agent 能力增强路线图

> 日期：2026-03-20
> 目标：从“可用的双后端 Agent 系统”演进到“接近 Claude Code / 协作型 Cowork Agent 的工作型 Agent”
> 范围：聚焦 Agent 能力本身，不展开渠道接入、品牌改造、基础工程迁移

## 一、当前基线

当前系统已经具备这些基础：

- 统一运行时：`AgentRuntime -> AgentRouter -> backend`
- 双正式后端：`litellm`、`claude_sdk`
- 核心会话能力：连续会话、session 持久化、`/new` `/stop` `/restart` `/help`
- 能力装配基础：skill、builtin tool、external MCP 已有共享 catalog
- Claude SDK 增强路径：session 复用、specialist agents、基础 handoff policy
- 过程可见性：progress、tool hint、Telegram 可见性
- 运行态诊断基础：capability summary、policy summary、session runtime trace

这说明系统已经不是“实验性 Agent”，而是一个稳定的 Agent 平台底座。

但如果目标是继续增强，做到更接近 Claude Code / Cowork 级别的工作型 Agent，当前还缺的是更高层能力，而不是更多零散工具。

## 二、目标状态

目标中的 Agent 不应只是：

- 会聊天
- 会调用工具
- 会读写代码

而应该是：

- 能接任务
- 能做计划
- 能持续执行
- 能调用专业 agent 协作
- 能记录上下文与过程
- 能解释为什么这样做
- 能稳定地产出结果

一句话：

**从“工具增强的助手”，升级成“有任务能力、协作能力、记忆能力、交付能力的工作 Agent”。**

## 三、核心能力缺口

从宏观角度看，当前主要还缺 6 类能力。

### 3.1 任务系统不足

现在的系统仍然偏“消息驱动”。  
它缺少一个真正的任务模型来承载：

- 任务状态
- 任务拆分
- 任务阻塞
- 恢复执行
- 结果验收

没有任务层，Agent 很难真正长期工作。

### 3.2 计划与执行仍未分层

当前有 planning 倾向，但还没有明确的：

- planner
- executor
- reviewer
- summarizer

复杂任务仍然更多依赖单次模型推理，而不是稳定的执行链条。

### 3.3 多 Agent 协作还只是基础形态

现在已经有：

- `spawn`
- Claude SDK specialist agents
- 基础 handoff / fallback trace

但还没有真正的协作协议：

- 谁负责什么
- 何时并行
- 结果如何汇总
- 谁对最终交付负责

### 3.4 记忆层次还不够完整

现在有 session、memory consolidation、workspace memory，但还没有明确分层成：

- session memory
- project memory
- user memory
- task memory

也缺少对 recall / synthesis / decay / validation 的治理。

### 3.5 可解释性和观测仍然偏基础

现在已经有 trace 和 summary，但还不足以完整回答：

- 为什么这次 handoff
- 为什么没用某个 skill
- 为什么用了这个 MCP
- 当前任务到底卡在哪
- 本轮执行链条怎么走的

### 3.6 交付导向仍不够强

当前系统更擅长“完成一次回答”，还不够擅长“完成一次工作交付”。

还需要更强的：

- 结果结构化
- 验证
- 自检
- 回归
- 失败重试
- 交付总结

## 四、阶段路线图

建议后续按 4 个大阶段推进，每阶段都以一个清晰目标为中心，不做碎片式推进。

---

## 阶段 A：任务系统化

### 目标

把 Agent 从“消息驱动”升级成“任务驱动”。

### 本阶段重点

- 建立任务对象和生命周期
- 明确任务状态：pending / running / blocked / done / failed
- 支持任务级持久化
- 支持长任务恢复
- 为每个任务定义结果与验收摘要

### 为什么优先级最高

这是后续所有增强的承载层。

没有任务系统，后面的：

- 协作
- 计划执行分层
- 长时记忆
- 交付追踪

都会落在消息流里，越来越乱。

### ROI

**最高**

原因：

- 直接提升 Agent 做复杂工作的稳定性
- 直接提升可恢复性
- 直接为多 Agent 协作提供容器
- 后续几乎所有高级能力都依赖它

### 风险

**中高**

风险点：

- 容易与现有 session/message 模型重叠
- 如果任务层和会话层职责不清，会变成两套状态机
- 要避免一开始设计得过重

### 控制方式

- 先做轻量任务模型，不碰底层消息协议
- 把 session 视为“对话载体”，把 task 视为“工作载体”
- 先支持主任务，再扩到子任务

### 完成标志

- 系统能明确知道“当前在做什么任务”
- 一个任务可以跨多轮消息持续执行
- Agent 可以恢复未完成任务

---

## 阶段 B：计划/执行/复核分层

### 目标

让 Agent 具备稳定的工作流程，而不是全靠单轮推理。

### 本阶段重点

- 引入 planner / executor / reviewer / summarizer 四层职责
- 复杂任务先计划，再执行，再复核，再汇报
- 支持明确的阶段切换
- 支持结果不达标时回到执行或计划层

### 为什么排第二

任务系统完成后，最先该补的是执行流程本身。  
否则任务只是“有了壳”，里面还是原来的随即式推理。

### ROI

**很高**

原因：

- 直接提升复杂任务完成率
- 直接降低“做了一半跑偏”的概率
- 为代码任务、研究任务、自动化任务提供统一流程骨架

### 风险

**中**

风险点：

- 流程太重会拖慢简单任务
- 分层做得不好会显得机械

### 控制方式

- 只对复杂任务启用完整流程
- 简单任务保持直达模式
- 用 runtime policy 判断是否升级为多阶段执行

### 完成标志

- Agent 能区分简单任务和复杂任务
- 复杂任务能稳定经历“计划 -> 执行 -> 复核 -> 交付”
- 失败时能明确回退到上一个阶段

---

## 阶段 C：多 Agent 协作化

### 目标

把现在的 handoff / spawn，升级成真正的协作型多 Agent 体系。

### 本阶段重点

- 定义主 Agent、specialist agent、background worker 的职责
- 定义任务拆分协议
- 定义并行执行协议
- 定义结果回流和汇总协议
- 定义 owner 和 final responder

### 为什么排第三

没有任务系统和流程分层，协作只会增加混乱。  
到这个阶段再做，底座会稳很多。

### ROI

**高**

原因：

- 可以明显提升复杂任务吞吐量
- 适合更接近 Cowork 形态
- 能释放 Claude SDK native agents/handoffs 的价值

### 风险

**高**

风险点：

- 子 agent 越多，状态同步越难
- trace 和 ownership 不清会导致难以排查
- 用户会感知到“系统变复杂”

### 控制方式

- 明确只有主 Agent 对用户负责
- 子 Agent 只负责局部交付
- 每次 handoff / spawn 必须留下决策与结果 trace

### 完成标志

- 能稳定拆任务给 specialist / background worker
- 子 Agent 的结果能被主 Agent 汇总
- 用户能理解系统当前是谁在做什么

---

## 阶段 D：记忆、观测、交付一体化

### 目标

把 Agent 做成真正可持续工作的系统。

### 本阶段重点

- 建立分层记忆
  - session memory
  - task memory
  - project memory
  - user memory
- 完善运行态观测
  - session trace
  - task trace
  - handoff trace
  - tool / skill / MCP trace
- 强化交付链条
  - 验证
  - 自检
  - 失败重试
  - 结果摘要

### 为什么放最后

这阶段是把 Agent 从“能工作”变成“能长期可靠工作”的关键，但它依赖前面三层都已经成型。

### ROI

**中高**

原因：

- 单项提效不如任务系统直接
- 但对稳定性、可信度、可维护性提升极大

### 风险

**中**

风险点：

- 记忆容易膨胀和污染
- trace 容易过多、过吵
- 验证链条如果设计过重，会拖慢交付

### 控制方式

- 记忆分层存储，明确保留策略
- trace 采用摘要 + 细节分层
- 验证按任务类型分级启用

### 完成标志

- Agent 不只会做事，还会留下可恢复、可解释、可追溯的工作记录
- 复杂任务可以长期持续推进
- 系统具备接近 Cowork 型 Agent 的基础形态

## 五、优先级排序

如果从 **ROI / 依赖关系 / 风险可控性** 综合排序，建议就是：

1. 阶段 A：任务系统化
2. 阶段 B：计划/执行/复核分层
3. 阶段 C：多 Agent 协作化
4. 阶段 D：记忆、观测、交付一体化

排序原因很简单：

- A 是底座
- B 是工作流骨架
- C 是协作能力释放
- D 是长期稳定性和可信度增强

## 六、投入与回报判断

### 最值得先做的

**任务系统化**

因为它对后续所有能力都是乘数效应。

### 最容易做成“看起来很强，实际上很乱”的

**多 Agent 协作化**

如果前面的任务系统和流程分层没做好，协作会先变成复杂度放大器。

### 最容易被低估，但长期最关键的

**记忆、观测、交付一体化**

因为系统越强，越需要：

- 被理解
- 被追踪
- 被恢复
- 被信任

## 七、建议的执行方式

后续实施建议继续保持现在这种模式：

- 一次只推进一个大阶段
- 每个阶段结束必须做定向测试 + 全量测试
- 每个阶段都要有明确完成标志
- 不在一个阶段里混入不相关的“顺手优化”

建议推进节奏：

- 先完成阶段 A
- 再完整做阶段 B
- 再做阶段 C
- 最后做阶段 D

不要改成：

- 同时开很多平行小项
- 看到问题就顺手补一点
- 一边做架构一边做无关体验优化

这会重新回到碎片化推进。

## 八、结论

如果你的目标只是“更强一点”，那继续补几个 tool、skill、trace 就够了。

如果你的目标是：

**让 xbot 真正成长为接近 Claude Code / Cowork 的工作型 Agent**

那后面最关键的，不是继续堆工具，而是补齐这四层：

- 任务
- 流程
- 协作
- 长期记忆与交付

这四层补起来之后，系统才会从“会干活的 Agent”进入“能持续承担工作”的阶段。
